ADR-002 · claims-app · Slice 1

Core data model: entities and tooling choices

Five decisions made before writing any models — each affecting every slice that follows.

Status
Accepted
Date
June 2026
Slice
1 — Data Model
Stack
Python · SQLAlchemy · Alembic · PostgreSQL
Context

Slice 1 requires a PostgreSQL schema to represent the core insurance domain: customers, policies, and claims. Before writing any models we needed to settle five tooling and design choices that affect every future slice — primary key strategy, ORM selection, migration tooling, status field representation, and entity scope.

These are not independent decisions: the choice of sync SQLAlchemy, for example, is contingent on deferring async concerns to a later slice. Documenting them together makes the dependency visible.

Decisions
1
UUID primary keys (not integer sequences)
UUID v4 generated in Python
Options
UUID v4 (Python) UUID (Postgres gen_random_uuid) Integer auto-increment
Rationale
UUIDs are safe to expose in URLs — no sequential enumeration risk. They work across distributed systems without coordination and are the standard in insurance and finance contexts where record IDs may appear in external documents or regulatory filings.
↕
UUID columns are larger (16 bytes vs 4) and slightly slower to index than integers. Acceptable at this scale.
2
SQLAlchemy ORM, synchronous
SQLAlchemy 2.0 ORM · DeclarativeBase · Mapped annotations
Options
SQLAlchemy ORM (sync) SQLAlchemy ORM (async) SQLAlchemy Core Raw psycopg2 SQL
Rationale
The ORM keeps model definitions readable and Pythonic, reducing boilerplate for standard CRUD. Synchronous chosen over async to minimise complexity in Slice 1 — async SQLAlchemy requires a different session lifecycle and is not needed until FastAPI concurrency becomes a bottleneck.
↕
Sync SQLAlchemy will require replacement if FastAPI concurrency becomes a bottleneck under real load. A known rework cost, accepted now in exchange for simplicity.
3
Alembic for schema migrations
Alembic with autogenerate
Options
Alembic (autogenerate) Manual SQL scripts Recreate DB on change No migration tooling
Rationale
Alembic is the standard companion to SQLAlchemy. Autogenerate derives the migration file from the models, reducing human error. A full migration history from day one is non-negotiable for any system that will hold real data — each change is auditable and reversible.
↕
Autogenerated migrations must always be reviewed before running — Alembic can miss certain changes (e.g. column renames) and may generate an empty file if models are not correctly imported in env.py.
4
Status fields as plain strings
String column · validation deferred to API layer
Options
Plain String Postgres native ENUM Python enum.Enum → PG ENUM Integer code
Rationale
Postgres ENUMs require a migration to add a new value, creating friction as domain logic evolves. Plain strings are easier to extend; validation of allowed values will be enforced at the API layer via Pydantic in a later slice, keeping the database schema flexible.
↕
Invalid status values are not rejected at the database level — a bug in the application layer could write bad data. This is acceptable given Pydantic validation will be added in Slice 2.
5
Three entities in scope (Customer, Policy, Claim)
Customer · Policy · Claim
Options
3 entities (this slice) 4 entities incl. ClaimDocument
Rationale
ClaimDocument was deferred to a later slice. Keeping the initial migration simple ensures Alembic setup can be validated cleanly, and the scope stays focused on the core relational model without introducing file-handling concerns.
Consequences
Positive
  • Sync SQLAlchemy and plain-string status fields deliberately defer complexity, keeping Slice 1 lean and unblocking progress toward the AI subsystem — the primary focus of this project.
  • Alembic gives the schema a full migration history from day one, making future changes auditable and reversible rather than manual and risky.
Negative / trade-offs
  • Sync SQLAlchemy will require replacement if FastAPI concurrency becomes a bottleneck under real load — a known rework cost accepted now for simplicity.
  • ClaimDocument deferred to a later slice; any features depending on document storage cannot be built until that entity is added.
Open questions
?
When does sync SQLAlchemy become a problem? To be revisited when the API layer is under load testing in a later slice.
?
Should status values be validated at the model layer or only at the API layer? Currently deferred to the API layer (Pydantic). To be decided in Slice 2 when the first endpoints are built.
