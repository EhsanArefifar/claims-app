§3 · Guiding Principles
Principles
Six rules that govern every decision in this plan. When in doubt, return here.
1
Build one real thing; learn on demand.
Not a syllabus to consume — a project to ship, pulling theory in when the build demands it. Passive consumption without production is the failure mode.
2
You build it.
Use Claude Code to go faster, but understand every line and be able to defend it. No hiding gaps behind orchestration.
3
ADRs as a habit.
Every meaningful decision gets written down. Target 15–25 over the year. The reasoning is the portfolio — not the code itself.
4
Hype-resistance is the product.
Knowing when not to use an agent / AI is rarer and more valuable than building the flashiest demo. Every AI feature gets a cost/benefit review.
5
Hands-on-credible, not senior dev.
The target is fluency — building a POC unaided and defending technical decisions — not a decade of engineering depth. Calibrate ambition to the target.
6
Anchor on real work where possible.
Modernization especially is fed by the day job. Every NTT Data task is also a learning artifact when documented correctly.

§5 · Flagship Project
Insurance Claims App
The single greenfield project that drives all SWE and AI learning. Front-to-back, built by you.
Scope
Build a comprehensive insurance claims application — front and back, from scratch. Greenfield, built by you. The core (non-AI) app comes first; then a genuine AI subsystem (multiple features, not a single flavour). Modernization stays a separate, later project fed by your team work.

Core entities: Customer, Policy, Claim (add ClaimDocument later).
AI Subsystem — Phase D (layered onto core app)
01
Document Extraction
Claude returns structured fields from uploaded claim documents.
02
Claim Triage
Severity and risk flagging over claim and policy data.
03
Policyholder Assistant
RAG over policy docs + tool-calls to your own API for live status.
04
Adjuster Summary
On-demand claim timeline and narrative summary generation.
05
Agentic Claim Processing
Multi-step agent: validate → check coverage (tool) → request missing info → draft recommendation. The showpiece portfolio artifact.
Showpiece
06
Evaluation Harness
Quantitative measurement of AI feature quality and regression testing.

Each AI feature maps to a CCA-F exam domain — the build doubles as certification preparation evidence.

§13 · Setup
Development Environment
Designed for the corporate laptop constraint. Everything runs in the browser.
GitHub Codespaces
Full VS Code in the browser on the corporate laptop (nothing to install) and via desktop VS Code from home — same environment, same devcontainer, both places.
Cost: 120 free core-hours/month per personal account, then ~$0.18/hr (a few dollars in heavy months).
Neon Serverless PostgreSQL
Free tier. Persists across Codespace rebuilds. Identical connection string from both machines — no local Postgres needed. Alternative: devcontainer Postgres for fully local operation.
Python / FastAPI / React stack
uv for package management; SQLAlchemy + Alembic for the DB layer; shadcn/ui + Tailwind for the frontend. All installed in the devcontainer.
AWS App Runner + GitHub Actions
Deployment target for Slice 6. App Runner removes cluster management overhead; Actions provides the CI pipeline. Avoids ECS/Fargate complexity while still being genuine AWS.
Caveats & Fallbacks
Confirm the corporate network allows GitHub/Codespaces before starting Slice 0. If blocked: use AWS CodeCatalyst (AWS-native, doubles as SAP practice) or build from home only.
