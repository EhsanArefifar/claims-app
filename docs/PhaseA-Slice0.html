What we built

The repo
claims-app on GitHub

A Git repository that holds all your code, config, and infrastructure-as-code. Everything is version-controlled from day one — no "I'll add it to Git later."

The environment
GitHub Codespaces

A full Linux development machine that runs in the cloud, inside your browser. No installs on your corporate laptop. Same environment from any device.

The container config
devcontainer

A recipe (.devcontainer/) that tells Codespaces exactly what to install and how to wire services together. Anyone who opens the repo gets an identical machine.

The skeleton app
FastAPI + Postgres

A minimal Python web server (main.py) with one /health endpoint, wired to a real Postgres database. Proof that the whole stack connects.

How it actually works

Your machine Browser / VS Code
──▶
GitHub cloud Codespace VM
──▶
Inside the VM app container
──▶
Sidecar container Postgres :5432
① Your browser connects to a GitHub-hosted VM via HTTPS — no port forwarding through your router.
② The VM runs two Docker containers (defined in docker-compose.yml): one for your Python app, one for Postgres.
③ The app container reaches Postgres via the hostname db — Docker's internal network, not localhost.
④ Codespaces proxies port 8000 to a public HTTPS URL so your browser can reach the app. Must be set to Public to work without GitHub auth headers.

What we created and why

repo layout
claims-app/
├── .devcontainer/
│   ├── devcontainer.json   # tells Codespaces what to install & run
│   └── docker-compose.yml  # defines app + db containers
├── main.py              # FastAPI app — one /health endpoint
├── requirements.txt     # Python packages to install
└── .env.example         # template for env vars (never commit real .env)
docker-compose.yml — what it actually does
app service:
  image: python:3.12             # official Python container
  DATABASE_URL → db:5432         # passes connection string to FastAPI
  depends_on: db                 # waits for Postgres before starting

db service:
  image: postgres:16             # official Postgres container
  POSTGRES_USER: claims          # creates this user on first start
  POSTGRES_DB: claimsdb          # creates this database on first start
  volume: postgres-data          # data survives container restarts

How to start working each day

1
Open your Codespace
Go to github.com → your repo → Code → Codespaces → Resume. If it's been idle it may take 30–60 seconds to warm up. Don't create a new one — resume the existing one to preserve your state.
2
Start the app server
In the terminal:
uvicorn main:app --reload --host 0.0.0.0
The --reload flag means the server restarts automatically whenever you save a file. The --host 0.0.0.0 makes it reachable through Codespaces' proxy.
3
Check port visibility
In VS Code, click the Ports tab at the bottom. Port 8000 must show Public. If it says Private, right-click → Change Port Visibility → Public.
4
Open Swagger UI
In the Ports tab, click the globe icon next to port 8000, then add /docs to the URL. You should see the interactive API docs.
https://fantastic-funicular-xxxxx-8000.app.github.dev/docs
5
Verify Postgres (optional daily check)
Open a second terminal tab and run:
psql postgresql://claims:claims@db:5432/claimsdb -c "SELECT 1;"

----------
        1

Using across different machines

Corporate laptop (browser only)
  • Open github.com in any browser
  • Navigate to your repo
  • Code → Codespaces → Resume existing codespace
  • VS Code opens fully in the browser tab
  • Nothing installed on the machine — firewall-safe
  • Same environment as always
Home machine (desktop VS Code)
  • Install VS Code + GitHub Codespaces extension
  • Sign in with your GitHub account
  • Cmd/Ctrl+Shift+P → "Connect to Codespace"
  • Pick your existing codespace
  • Full desktop VS Code experience, same cloud VM
  • Better for long sessions — lower latency feel
Key point

There is only one codespace — the same VM running in GitHub's cloud. Whether you connect from your corporate laptop browser or your home VS Code, you're editing the same files on the same machine. Your running server, your Postgres data, your terminal history — all shared.

Free hours — how to check & understand them

What you get free

  • 120 core-hours / month on a personal account
  • A 2-core machine = 60 actual hours of active use
  • 15 GB storage free (your code + Postgres data)
  • Resets on your GitHub billing date each month
  • After free tier: ~$0.18/hr (2-core)

Best practices to preserve hours

Save

Stop, don't delete. When you finish a session, stop the codespace (Code → Codespaces → Stop) rather than closing the tab. A stopped codespace uses zero compute hours. Only a running codespace consumes your allocation.

Save

Set auto-stop timeout. Go to github.com/settings/codespaces and set the inactivity timeout to 30 minutes. If you walk away and forget, it stops itself. Default is 30 min but worth confirming.

Save

Use 2-core machines. The default 2-core machine is sufficient for this entire project through Phase A and D. Don't upgrade unless you hit a real bottleneck — a 4-core machine halves your free hours.

Note

Storage is separate from compute. Even a stopped codespace uses storage. The Postgres volume (postgres-data) persists and counts toward your 15 GB. Not a concern at this stage, but worth knowing.

Info

Prebuilds exist but aren't needed yet. GitHub can pre-build your devcontainer so it opens instantly, but this costs extra storage. Skip for now — the 2–3 min cold start is fine at this stage.

Note

Don't commit .env files. The .env.example file is safe to commit — it's a template. The real .env file (with actual credentials) must stay in .gitignore. Even for a private repo, this is the habit to build now.
