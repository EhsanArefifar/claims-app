Concept

Big picture — what is Slice 1 doing?

Before writing any API endpoints, you need a place to store data. That means designing the shape of your data (the model), setting up the database (PostgreSQL), teaching Python to talk to it (SQLAlchemy), and building a system to evolve it safely over time (Alembic).

Data model

The blueprint — what things exist, what fields they have, how they relate. Like an architect's floor plan before any concrete is poured.

PostgreSQL

The actual storage system. Holds your data in structured tables on disk and knows how to find, save, and update it reliably.

SQLAlchemy

A Python library letting you describe tables as Python classes and talk to the database in Python — no raw SQL required.

Alembic

A migration tool. When you change your model later, Alembic generates the SQL scripts to update the database without losing data.

ADR

Architectural Decision Record. A short doc explaining why you made a key technical choice — your portfolio evidence and future-self's best friend.

Analogy

Imagine building a filing cabinet system for an insurance office. The data model is the design for how many drawers and what labels go inside. PostgreSQL is the physical cabinet. SQLAlchemy is the office manager who files and retrieves things for you. Alembic is the system for reorganising the cabinet when you need a new drawer — without losing anyone's files.

Concept

Data model

A data model defines what things your app needs to remember, what information each thing has (fields), and how those things relate to each other. Think of it as the agreed structure everyone works from.

Analogy

A spreadsheet has column headers — Name, Email, Phone. That's a data model. A relational data model is like multiple related spreadsheets where rows in one sheet can point to rows in another.

Concept

PostgreSQL

PostgreSQL stores data in tables (rows and columns) on disk and ensures reliability even if the server crashes mid-write. Your FastAPI app connects to it using a connection URL stored in an environment variable.

Mental model

Think of Postgres as Excel — but it runs as a server, handles thousands of users at once, enforces rules (you can't have a policy without a customer), and never loses data.

Tables

Each entity becomes a table. Columns are the fields, rows are the actual records.

Constraints

Rules enforced by the DB — UNIQUE (no two emails the same), NOT NULL (must have a value), FOREIGN KEY (that customer_id must actually exist).

SQL

The language you use to talk to it. SELECT gets data, INSERT adds it, UPDATE changes it. Alembic and SQLAlchemy write most of this for you.

The connection URL format

# format
postgresql://user:password@host:5432/dbname

# in your devcontainer
DATABASE_URL=postgresql://postgres:postgres@db:5432/claims_db
💡
Why not docker ps? Inside a Codespace you are already running inside the devcontainer — the Docker CLI is not available from there. Use pg_isready -h db -p 5432 instead, which talks directly to the Postgres service over the internal network.
Concept

SQLAlchemy — Python talks to Postgres

SQLAlchemy is an ORM (Object Relational Mapper). It lets you define your tables as Python classes and translates your Python code into SQL automatically — safer and less error-prone than writing raw SQL strings.

ORM analogy

Instead of writing INSERT INTO customers VALUES (...) by hand, you write db.add(Customer(name="María")). SQLAlchemy handles the rest. Same result, much safer.

Engine

The connection to the database. Created once at startup from DATABASE_URL. Manages the connection pool.

Session

A unit of work. Open a session, make changes, commit. If anything fails, roll back — like a transaction.

Base + Models

Base is the parent class all your models inherit from. Each model (Customer, Policy, Claim) maps to a table.

What a model looks like

class Customer(Base):
    __tablename__ = "customers"

    id = mapped_column(UUID(as_uuid=True), primary_key=True, default=uuid.uuid4)
    full_name = mapped_column(String, nullable=False)
    email = mapped_column(String, unique=True, nullable=False)
    created_at = mapped_column(DateTime, default=datetime.utcnow)

Each mapped_column() call defines one field. The keyword arguments (primary_key, nullable, unique) become database constraints.

Concept

Alembic — managing change safely

Once your app is live, you can't delete and recreate the database every time you change a model — you'd lose all data. Alembic tracks every schema change and generates SQL scripts to apply them step by step.

Git for your database

Think of Alembic like Git for your database schema. Each migration is a commit — it describes exactly what changed and can be applied or reversed. You can see the full history of how the schema evolved.

autogenerate

Alembic compares your SQLAlchemy models against the current database state and writes the migration for you. You review, then run.

upgrade head

Apply all pending migrations up to the latest version. "Head" means the tip of the migration chain.

downgrade

Each migration also has an undo script. If something goes wrong, you can roll the schema back to a previous state.

What a generated migration looks like

def upgrade() -> None:
    op.create_table(
        'customers',
        sa.Column('id', postgresql.UUID(), nullable=False),
        sa.Column('full_name', sa.String(), nullable=False),
        sa.Column('email', sa.String(), nullable=False),
        sa.PrimaryKeyConstraint('id'),
        sa.UniqueConstraint('email')
    )

def downgrade() -> None:
    op.drop_table('customers')  # how to undo it
⚠️
Common gotcha: if the run_migrations_online function in env.py is missing the with connectable.connect() block, Alembic silently generates an empty migration — upgrade() just says pass. Always read the generated file before running it.

Step 1

Install dependencies

Check whether a virtual environment is already active — you'll see (.venv) at the start of your terminal prompt. If not:

python -m venv .venv
source .venv/bin/activate
💡
Why a venv? An isolated copy of Python with its own packages. Without one, every package goes into the system Python — shared across all projects, causing version conflicts. Good habit to build now even inside a Codespace.

Install the three packages:

pip install sqlalchemy alembic psycopg2-binary

Pin the versions automatically:

pip freeze > requirements.txt
💡
pip freeze lists every installed package and its exact version. The > writes that to the file — captures real installed versions, not guessed ones.

Step 2

Create app/db.py

This file does three things: creates the engine (connection to Postgres), creates SessionLocal (how you open/close transactions), and creates Base (parent class all models inherit from).

mkdir -p app
touch app/db.py

Paste this into app/db.py:

import os
from sqlalchemy import create_engine
from sqlalchemy.orm import sessionmaker, DeclarativeBase

DATABASE_URL = os.environ["DATABASE_URL"]

engine = create_engine(DATABASE_URL)
SessionLocal = sessionmaker(bind=engine)

class Base(DeclarativeBase):
    pass

app/models/customer.py

import uuid
from datetime import datetime
from sqlalchemy import String, DateTime
from sqlalchemy.orm import Mapped, mapped_column
from sqlalchemy.dialects.postgresql import UUID
from app.db import Base

class Customer(Base):
    __tablename__ = "customers"

    id: Mapped[uuid.UUID] = mapped_column(UUID(as_uuid=True), primary_key=True, default=uuid.uuid4)
    full_name: Mapped[str] = mapped_column(String, nullable=False)
    email: Mapped[str] = mapped_column(String, unique=True, nullable=False)
    phone: Mapped[str | None] = mapped_column(String, nullable=True)
    created_at: Mapped[datetime] = mapped_column(DateTime, default=datetime.utcnow, nullable=False)

app/models/policy.py

import uuid
from datetime import datetime, date
from sqlalchemy import String, DateTime, Date, ForeignKey
from sqlalchemy.orm import Mapped, mapped_column
from sqlalchemy.dialects.postgresql import UUID
from app.db import Base

class Policy(Base):
    __tablename__ = "policies"

    policy_number: Mapped[str] = mapped_column(String, unique=True, nullable=False)
    type: Mapped[str] = mapped_column(String, nullable=False)

    end_date: Mapped[date | None] = mapped_column(Date, nullable=True)

app/models/claim.py

import uuid
from datetime import datetime, date
from sqlalchemy import String, DateTime, Date, ForeignKey
from sqlalchemy.orm import Mapped, mapped_column
from sqlalchemy.dialects.postgresql import UUID
from app.db import Base

class Claim(Base):
    __tablename__ = "claims"

    id: Mapped[uuid.UUID] = mapped_column(UUID(as_uuid=True), primary_key=True, default=uuid.uuid4)
    policy_id: Mapped[uuid.UUID] = mapped_column(UUID(as_uuid=True), ForeignKey("policies.id"), nullable=False)
    reference_number: Mapped[str] = mapped_column(String, unique=True, nullable=False)
    status: Mapped[str] = mapped_column(String, nullable=False, default="submitted")
    description: Mapped[str | None] = mapped_column(String, nullable=True)
    incident_date: Mapped[date] = mapped_column(Date, nullable=False)
    filed_at: Mapped[datetime] = mapped_column(DateTime, default=datetime.utcnow, nullable=False)
    updated_at: Mapped[datetime] = mapped_column(DateTime, default=datetime.utcnow, onupdate=datetime.utcnow, nullable=False)

Step 4

Initialise Alembic

alembic init alembic

Edit alembic.ini — find and replace this line:

# find:
sqlalchemy.url = driver://user:pass@localhost/dbname

# replace with:
sqlalchemy.url = %(DATABASE_URL)s

Edit alembic/env.py — find the target_metadata = None block and replace it:

from app.models.customer import Customer  # noqa: F401
from app.models.policy import Policy      # noqa: F401
from app.models.claim import Claim        # noqa: F401
from app.db import Base

target_metadata = Base.metadata

Then replace the entire run_migrations_online function:

def run_migrations_online() -> None:
    import os
    connectable = create_engine(os.environ["DATABASE_URL"])

    with connectable.connect() as connection:
        context.configure(
            connection=connection,
            target_metadata=target_metadata,
        )
        with context.begin_transaction():
            context.run_migrations()
⚠️
Critical: the with connectable.connect() block is required. Without it Alembic creates the engine and returns silently — generating an empty migration with pass in upgrade().
Verifyalembic current → no error (may show "no current revision" — that's fine)
Step 5

Generate and run the migration

First, confirm Alembic can see your models:

python -c "from app.models.customer import Customer; from app.models.policy import Policy; from app.models.claim import Claim; from app.db import Base; print(list(Base.metadata.tables.keys()))"
Expect['customers', 'policies', 'claims']
🔴
If you see []: Alembic cannot see your models — go back and check the imports in env.py.
If you see ImportError: Python can't find the app package. Make sure you're in the project root and that app/__init__.py exists: touch app/__init__.py

If a previous empty migration exists, delete it first:

rm alembic/versions/*.py

Generate the migration:

alembic revision --autogenerate -m "initial schema"

Open the generated file in alembic/versions/ and confirm upgrade() has create_table calls for all three tables. Then run it:

alembic upgrade head

Verify the tables:

psql $DATABASE_URL -c "\dt"
Expectalembic_version · claims · customers · policies
